# 08 — systemd Deployment

This runbook covers running `hokorad` as a systemd service on Linux. It's the recommended deployment mode for bare-metal and VPS nodes.

## Two templates exist — use the hardened one

**Important.** Hokora ships two systemd unit files and they are not equivalent:

| Source | Location | Hardening |
|---|---|---|
| Static template | `systemd/hokorad.service.template` | Minimal (`PrivateTmp`, `ProtectSystem=strict`, `NoNewPrivileges`, `ReadWritePaths`) |
| Generated unit | `$DATA_DIR/systemd/hokorad.service` after `hokora init` | Fully hardened (adds `ProtectHome`, `ProtectKernel*`, `MemoryDenyWriteExecute`, `LockPersonality`, `RestrictNamespaces`, `StopTimeoutSec`, `UMask`, etc.) |

**Always use the generated unit**, not the static template. The static file is a minimal reference; the generator in `src/hokora/cli/init.py::_generate_systemd` is the supported hardened version.

## Install the generated unit

```bash
# hokora init has already written $DATA_DIR/systemd/hokorad.service for you.
# Copy it into systemd, reload, enable, start.

sudo cp ~/.hokora/systemd/hokorad.service /etc/systemd/system/hokorad.service
sudo systemctl daemon-reload
sudo systemctl enable --now hokorad

# Verify
systemctl status hokorad
journalctl -u hokorad -n 100 --no-pager
```

The generated unit runs as `User=hokora`. Create the user before enabling:

```bash
sudo useradd -r -s /usr/sbin/nologin -d /var/lib/hokora hokora
sudo chown -R hokora:hokora /var/lib/hokora       # if that's your data dir
```

If you installed with a different user, edit the `User=` line in the generated unit before copying.

## Hardened unit reference

For reference, the generated unit looks like this:

```ini
[Unit]
Description=Hokora Daemon
After=network.target
Wants=network-online.target

[Service]
Type=simple
User=hokora
ExecStart=/path/to/python -m hokora
Environment=HOKORA_CONFIG=/var/lib/hokora/hokora.toml
Restart=on-failure
RestartSec=5
StopTimeoutSec=30
UMask=0077

# Sandbox hardening
PrivateTmp=yes
ProtectSystem=strict
ProtectHome=yes
ProtectKernelTunables=yes
ProtectKernelModules=yes
ProtectKernelLogs=yes
ProtectControlGroups=yes
ProtectProc=invisible
RestrictNamespaces=yes
RestrictRealtime=yes
RestrictSUIDSGID=yes
LockPersonality=yes
MemoryDenyWriteExecute=yes
NoNewPrivileges=yes
ReadWritePaths=/var/lib/hokora

[Install]
WantedBy=multi-user.target
```

### What each hardening option does

| Option | Protects against |
|---|---|
| `PrivateTmp` | `/tmp` cross-process snooping |
| `ProtectSystem=strict` | Writes to the filesystem outside `/var`/`/run`/`/tmp` and explicit `ReadWritePaths` |
| `ProtectHome=yes` | Reads of `/home`, `/root`, `/run/user` |
| `ProtectKernelTunables` | `/proc/sys` writes |
| `ProtectKernelModules` | `init_module(2)` and friends |
| `ProtectKernelLogs` | `dmesg` and kernel ring buffer |
| `ProtectControlGroups` | cgroupfs writes |
| `ProtectProc=invisible` | `/proc` entries of other users' processes |
| `RestrictNamespaces` | `clone(2)` with `CLONE_NEW*` flags |
| `RestrictRealtime` | `SCHED_FIFO`/`SCHED_RR` |
| `RestrictSUIDSGID` | `chmod u+s`, `chmod g+s` |
| `LockPersonality` | `personality(2)` mutation |
| `MemoryDenyWriteExecute` | W^X violations (JIT stacks etc.) |
| `NoNewPrivileges` | `setuid`/`setgid`/capabilities gain |
| `UMask=0077` | Files created by the service default to mode 0o700 |

If your Python interpreter lives outside `/usr` or `/opt`, or your data dir is outside the single `ReadWritePaths=`, override with a drop-in rather than weakening the unit globally:

```bash
sudo systemctl edit hokorad
# Add in the editor window:
[Service]
ReadWritePaths=/custom/path
```

## Relay service

For a relay-only node, copy the hardened unit generated by `hokora init --node-type relay` (which writes the same file with `--relay-only` appended to `ExecStart`). Example:

```ini
ExecStart=/opt/hokora/.venv/bin/python -m hokora --relay-only
```

## Log routing

The generated hardened unit does not redirect stdout/stderr — they go to journald. View with:

```bash
journalctl -u hokorad -f
journalctl -u hokorad --since "1 hour ago"
journalctl -u hokorad -p warning     # warnings and above
```

If you prefer an on-disk log file (e.g. for fleet log shippers that read files), set `HOKORA_LOG_TO_STDOUT=false` (default) and the daemon will write a rotated log at `$DATA_DIR/hokorad.log` (10 MB × 5 backups). For JSON structured logs, also set `HOKORA_LOG_JSON=true`.

## Rolling restarts

For fleet operators managing many nodes, stagger restarts so you don't blank-network a mesh:

```bash
for host in node-a node-b node-c; do
  ssh $host sudo systemctl restart hokorad
  sleep 60
done
```

Within the 60-second window, check the node came back healthy:

```bash
curl -sf http://$host:8421/health/ready
```

## Watchdog integration

Hokora's heartbeat file is transport-independent. To wire systemd's watchdog into it, use a small sidecar script:

```bash
#!/usr/bin/env bash
# /usr/local/bin/hokora-heartbeat-watchdog
HEARTBEAT=/var/lib/hokora/heartbeat
MAX_AGE=90   # seconds

while true; do
  if ! test -f "$HEARTBEAT"; then exit 1; fi
  age=$(( $(date +%s) - $(stat -c %Y "$HEARTBEAT") ))
  if (( age > MAX_AGE )); then exit 1; fi
  systemd-notify WATCHDOG=1
  sleep 30
done
```

Paired with a watchdog unit:

```ini
[Unit]
Description=Hokora Heartbeat Watchdog
BindsTo=hokorad.service
After=hokorad.service

[Service]
Type=notify
NotifyAccess=main
WatchdogSec=90
ExecStart=/usr/local/bin/hokora-heartbeat-watchdog
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

A stale heartbeat will now cause systemd to restart the sidecar (which under `BindsTo=` also bounces `hokorad`). Not wired by default — opt in if you want it.

## Fleet operator notes

- Distribute the hardened unit (or your own override) via your config management tool; do not rely on `hokora init` running on every host.
- Template the `ExecStart`, `User`, `ReadWritePaths`, and `Environment` values for your environment.
- Log to journald and ship with your existing journald-aware shipper (vector, fluent-bit, Grafana Agent). JSON mode (`HOKORA_LOG_JSON=true`) makes downstream parsing trivial.
- Alert on `systemctl is-failed hokorad` as well as the heartbeat freshness metric exposed over `/api/metrics`.
